## 상세 개선 내용

### 1. 중복 컴포넌트 제거 — `articles-client.tsx`

**문제**

`HighlightText` 컴포넌트가 파일 내에 두 번 정의되어 있었음.

- 모듈 레벨(38번째 줄): `function HighlightText(...)` — 올바른 위치
- `ArticlesClient` 함수 내부(137번째 줄): `const HighlightText = (...)` — 중복

컴포넌트 내부에 정의된 컴포넌트는 부모가 렌더링될 때마다 새로운 함수 참조가 생성되어 React가 기존 DOM을 unmount 후 remount함. 결과적으로 불필요한 레이아웃 재계산과 DOM 재생성이 발생.  
또한 내부 정의본은 모듈 레벨의 `ESCAPE_REGEXP` 상수 대신 인라인 정규표현식을 사용해 미세한 불일치도 존재했음.

**개선**

내부 중복 정의(35줄) 완전 제거. 모듈 레벨 `HighlightText`만 사용.

```diff
- // Highlighter component
- const HighlightText = ({
-   text,
-   highlight,
- }: {
-   text: string;
-   highlight: string;
- }) => {
-   if (!highlight.trim()) {
-     return <span>{text}</span>;
-   }
-   const safeHighlight = highlight.replace(/[.*+?^${}()|[\]\\]/g, "\\$&");
-   const regex = new RegExp(`(${safeHighlight})`, "gi");
-   const parts = text.split(regex);
-   return (
-     <span>
-       {parts.map((part, i) =>
-         regex.test(part) ? (
-           <mark key={i} ...>{part}</mark>
-         ) : (
-           <span key={i}>{part}</span>
-         ),
-       )}
-     </span>
-   );
- };
```

### 2. `any` 타입 제거 — `home-client.tsx`

**문제**

`PostMetadata` 타입에는 `views`, `likes`, `thumbnail_url` 필드가 없어 정렬 콜백에서 `any`로 캐스팅하고, 렌더링 시 `(post as any).views`를 사용하고 있었음.

```ts
// 이전
posts.toSorted((a: any, b: any) => { ... })
{(post as any).views || 0}
{(post as any).likes || 0}
```

**개선**

`PostWithMetrics` 로컬 타입을 정의하여 컴포넌트 props와 내부 로직을 타입 안전하게 변경.

```ts
// 추가된 타입
type PostWithMetrics = PostMetadata & {
  views?: number;
  likes?: number;
  thumbnail_url?: string | null;
};

// props 타입 변경
export function HomeClient({
  posts,
}: {
  posts: PostWithMetrics[];  // PostMetadata[] → PostWithMetrics[]
  ...
}) {
  // toSorted 타입 추론 가능
  const sortedPosts = useMemo(() =>
    posts.toSorted((a, b) => { ... }),  // (a: any, b: any) 제거
    [posts, sortType]
  );

  // 렌더링 캐스트 제거
  {post.views || 0}   // (post as any).views → post.views
  {post.likes || 0}   // (post as any).likes → post.likes
```

### 3. `any` 타입 제거 — `app/page.tsx`

**문제**

Supabase 조인 결과 처리 시 `any` 캐스팅과 `HomeClient`로 전달 시 `as any` 사용.

```ts
// 이전
post.post_tags?.map((pt: any) => pt.tags?.name).filter(Boolean)
<HomeClient posts={formattedPosts as any} ... />
```

**개선**

조인 결과에 명시적 인라인 타입 적용. `filter`에 타입 가드 추가로 반환 타입 `string[]` 보장.

```ts
const postTags = (
  post.post_tags as Array<{ tags: { name: string } | null }> ?? []
).map((pt) => pt.tags?.name).filter((n): n is string => Boolean(n));

// as any 캐스트 제거 (HomeClient가 PostWithMetrics[]를 받도록 수정됐으므로)
<HomeClient posts={formattedPosts} ... />
```

### 4. 동적 SEO 메타데이터 + 이미지 최적화 — `posts/[slug]/page.tsx`

#### 4-a. `generateMetadata` 추가

**문제**

포스트 상세 페이지에 메타데이터 생성 함수가 없어 모든 포스트가 루트 레이아웃의 기본 제목("OKOJIN의 블로그")을 공유. 검색 엔진 결과와 SNS 공유 시 콘텐츠별 제목·설명·OG 이미지가 표시되지 않음.

**개선**

`generateMetadata`를 추가해 포스트별 동적 메타데이터 생성. 필요한 컬럼만 select하여 데이터 전송량 최소화.

```ts
export async function generateMetadata(props: {
  params: Promise<{ slug: string }>;
}): Promise<Metadata> {
  const { slug } = await props.params;
  const { data: post } = await supabase
    .from("posts")
    .select("title, description, thumbnail_url")  // 필요한 필드만
    .eq("slug", slug)
    .single();

  if (!post) return { title: "Not Found" };

  return {
    title: post.title,  // layout.tsx template과 결합 → "제목 | OKOJIN의 블로그"
    description: post.description,
    openGraph: {
      title: post.title,
      description: post.description,
      type: "article",
      ...(post.thumbnail_url && {
        images: [{ url: post.thumbnail_url, width: 1200, height: 630 }],
      }),
    },
  };
}
```

#### 4-b. 배경 이미지 `next/image` 전환

**문제**

포스트 헤더의 썸네일 배경을 CSS `backgroundImage`로 렌더링. 이 방식은 Next.js Image Optimization 파이프라인을 거치지 않아 WebP 자동 변환, 사이즈 최적화, LCP 우선 로딩 없음.

```tsx
// 이전
<div
  className="absolute inset-0 bg-cover bg-center bg-no-repeat transition-transform duration-1000 hover:scale-105"
  style={{ backgroundImage: `url(${post.thumbnail_url})` }}
/>
```

**개선**

`next/image`의 `fill` + `priority` prop 사용으로 교체. CDN 최적화된 WebP 이미지 제공 및 LCP 요소 우선 로딩.

```tsx
<Image
  src={post.thumbnail_url}
  alt={post.title}
  fill
  className="object-cover transition-transform duration-1000 hover:scale-105"
  priority
/>
```

**효과:**
- WebP/AVIF 자동 포맷 변환으로 이미지 크기 30~50% 절감
- `priority`로 LCP(Largest Contentful Paint) 개선
- Vercel CDN 캐싱 활용

#### 4-c. `any` 타입 제거

포스트 태그 조인 결과에 명시적 타입 적용. (app/page.tsx와 동일한 패턴)

```ts
// 이전
post.post_tags?.map((pt: any) => pt.tags?.name).filter(Boolean)

// 이후
(post.post_tags as Array<{ tags: { name: string } | null }> ?? [])
  .map((pt) => pt.tags?.name)
  .filter((n): n is string => Boolean(n))
```

### 5. OpenGraph · title template 추가 — `layout.tsx`

**문제**

루트 메타데이터에 `title`, `description`만 설정. SNS 공유 시 OG 태그 없음, 하위 페이지에서 title 포맷 일관성 없음.

**개선**

- `title.template`으로 포스트 제목 포맷 통일 (`"게시글 제목 | OKOJIN의 블로그"`)
- `openGraph` 및 `twitter` 카드 메타데이터 추가

```ts
// 이전
export const metadata: Metadata = {
  title: "OKOJIN의 블로그",
  description: "프론트엔드 개발자의 아카이브 채널",
};

// 이후
export const metadata: Metadata = {
  title: {
    default: "OKOJIN의 블로그",
    template: "%s | OKOJIN의 블로그",
  },
  description: "프론트엔드 개발자의 아카이브 채널",
  openGraph: {
    title: "OKOJIN의 블로그",
    description: "프론트엔드 개발자의 아카이브 채널",
    type: "website",
    locale: "ko_KR",
  },
  twitter: {
    card: "summary_large_image",
  },
};
```

**효과:**
- 카카오톡·트위터·슬랙 등 SNS 공유 시 포스트 제목·설명·썸네일 자동 표시
- `generateMetadata` 반환값의 `title`이 template에 자동 합성됨

### 6. 이미지 CDN 캐시 TTL 설정 — `next.config.ts`

**문제**

`minimumCacheTTL` 미설정 시 Next.js 기본값(60초)이 적용되어 Supabase CDN 이미지가 빈번하게 재요청됨.

**개선**

블로그 썸네일처럼 거의 변경되지 않는 이미지는 1년 캐싱으로 CDN 적중률 향상 및 원본 서버 트래픽 절감.

```ts
// 이전
images: {
  remotePatterns: [...],
},

// 이후
images: {
  remotePatterns: [...],
  minimumCacheTTL: 60 * 60 * 24 * 365,  // 1년
},
```

**효과:** 동일 이미지의 반복 요청이 캐시에서 처리되어 이미지 로딩 속도 향상 및 Supabase 스토리지 대역폭 절감.

### 7. ISR 전환 — `algorithm/[slug]/page.tsx`

**문제**

`export const dynamic = "force-dynamic"` 설정으로 인해 모든 알고리즘 상세 페이지가 요청마다 서버에서 새로 렌더링됨.  
Shiki의 `codeToHtml`은 언어 파서를 로드하고 토큰화를 수행하는 CPU 집약적 작업인데, 이를 매 요청마다 두 번(`github-light`, `github-dark`) 실행하는 것은 불필요한 서버 부하.

**개선**

`revalidate = 3600`(1시간)으로 ISR 전환. 첫 요청에 렌더링 후 캐시에 저장, 1시간 경과 후 백그라운드에서 재검증.

```ts
// 이전
export const dynamic = "force-dynamic";

// 이후
export const revalidate = 3600;  // 1시간 ISR
```

**효과:**
- 동일 알고리즘 페이지 재방문 시 즉시 응답 (캐시 히트)
- Shiki 코드 하이라이팅 1시간에 1회만 실행
- 알고리즘 데이터는 빈번하게 변경되지 않으므로 1시간 TTL이 적합
