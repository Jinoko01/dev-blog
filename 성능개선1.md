## 1. `async-parallel` — 병렬 데이터 페칭

**파일:** `apps/web/src/app/page.tsx`  
**규칙:** 1.4 Promise.all() for Independent Operations (CRITICAL)

### 이전 (순차 실행, 2 round trip)
```ts
const { data: posts } = await supabase.from("posts").select(...);
const { data: metrics } = await supabase.from("post_metrics").select("*");
```

### 이후 (병렬 실행, 1 round trip)
```ts
const [{ data: posts }, { data: metrics }] = await Promise.all([
  supabase.from("posts").select(...),
  supabase.from("post_metrics").select("*"),
]);
```

**효과:** `posts`와 `post_metrics` 쿼리가 독립적이므로 병렬로 실행할 수 있습니다. 네트워크 왕복 횟수가 절반으로 줄어 페이지 초기 로딩 시간을 2배까지 단축할 수 있습니다.

## 2. `bundle-barrel-imports` — optimizePackageImports 설정

**파일:** `apps/web/next.config.ts`  
**규칙:** 2.1 Avoid Barrel File Imports (CRITICAL)

### 이전
```ts
const nextConfig: NextConfig = {
  images: { ... },
};
```

### 이후
```ts
const nextConfig: NextConfig = {
  experimental: {
    optimizePackageImports: ["lucide-react", "framer-motion"],
  },
  images: { ... },
};
```

**효과:** `lucide-react`는 1,583개 이상의 re-export를 가진 배럴 파일입니다. `optimizePackageImports`를 활성화하면 빌드 단계에서 자동으로 직접 임포트로 변환되어 콜드 스타트 200–800ms, 개발 서버 부팅 15–70% 절감 효과를 얻습니다.

## 3. `rerender-no-inline-components` — 컴포넌트 모듈 레벨 분리

**파일:** `apps/web/src/components/articles-client.tsx`  
**규칙:** 5.4 Don't Define Components Inside Components (MEDIUM)

### 이전
```tsx
export function ArticlesClient() {
  // ❌ 컴포넌트 내부에 정의 — 매 렌더마다 새 함수 참조 생성
  const HighlightText = ({ text, highlight }) => { ... };
  return <HighlightText ... />;
}
```

### 이후
```tsx
// ✅ 모듈 레벨 — 참조가 안정적으로 유지됨
function HighlightText({ text, highlight }) { ... }

export function ArticlesClient() {
  return <HighlightText ... />;
}
```

**효과:** 컴포넌트가 내부에 정의되면 부모가 렌더될 때마다 React가 해당 컴포넌트를 unmount/remount 처리하여 상태 초기화, DOM 재생성, 불필요한 레이아웃 재계산이 발생합니다.

## 4. `rendering-hoist-jsx` — 정적 JSX 호이스팅

**파일:** `apps/web/src/components/home-client.tsx`  
**규칙:** 6.3 Hoist Static JSX Elements (MEDIUM)

### 이전
```tsx
// Sparkline 렌더마다 새로운 ReactElement 생성
<defs>
  <linearGradient id="sparkFill" ...>...</linearGradient>
</defs>
```

### 이후
```tsx
// 모듈 레벨 상수 — 한 번만 생성
const SPARKLINE_DEFS = (
  <defs>
    <linearGradient id="sparkFill" ...>...</linearGradient>
  </defs>
);
```

**효과:** 컨텐츠가 변하지 않는 정적 JSX를 매번 재생성하지 않아 reconciliation 비용이 감소합니다.

## 5. `rendering-conditional-render` — 명시적 조건부 렌더링

**파일:** `articles-client.tsx`, `home-client.tsx`, `post-list.tsx`, `algorithm-list-client.tsx`  
**규칙:** 6.9 Use Explicit Conditional Rendering (MEDIUM)

### 이전 (`&&` 패턴)
```tsx
{topics.length > 0 && topics.map(...)}
{visibleCount < allPosts.length && <LoadingSpinner />}
```

### 이후 (삼항 연산자)
```tsx
{topics.length > 0 ? topics.map(...) : null}
{visibleCount < allPosts.length ? <LoadingSpinner /> : null}
```

**효과:** `&&`는 좌항이 `0`이나 `NaN` 같은 falsy primitive일 때 그 값 자체를 렌더링하는 React 특유의 함정이 있습니다. 명시적 삼항 연산자를 사용하면 이런 버그를 사전에 예방합니다.

## 6. `js-cache-storage` — localStorage 읽기 캐싱

**파일:** `apps/web/src/components/post-metrics.tsx`, `home-client.tsx`  
**규칙:** 7.5 Cache Storage API Calls (LOW-MEDIUM)

### post-metrics.tsx 이전
```ts
if (!localStorage.getItem(viewKey)) { ... }   // 1번째 읽기
if (localStorage.getItem(`liked_${slug}`)) { ... } // 2번째 읽기 (다른 key)
```

### 이후
```ts
const hasViewed = localStorage.getItem(viewKey);   // 한 번만 읽기
const hasLiked = localStorage.getItem(likedKey);   // 한 번만 읽기
if (!hasViewed) { ... }
if (hasLiked) { ... }
```

**효과:** localStorage 접근은 동기 I/O이며 메인 스레드를 블로킹합니다. 동일 key를 여러 번 읽지 않고 변수에 캐싱하면 불필요한 동기 작업이 줄어듭니다.

## 7. `js-tosorted-immutable` — 불변 정렬

**파일:** `apps/web/src/components/home-client.tsx`  
**규칙:** 7.13 Use toSorted() Instead of sort() for Immutability (LOW-MEDIUM)

### 이전
```ts
const copy = [...posts];
copy.sort((a, b) => ...);
return copy;
```

### 이후
```ts
return posts.toSorted((a, b) => ...);
```

**효과:** `toSorted()`는 원본 배열을 변이하지 않고 새 배열을 반환합니다. 코드가 간결해지고, React 의존성 비교 시 참조 투명성을 더 명확하게 유지합니다.

## 8. `js-hoist-regexp` — RegExp 호이스팅

**파일:** `apps/web/src/components/articles-client.tsx`  
**규칙:** 7.9 Hoist RegExp Creation (LOW-MEDIUM)

### 이전
```ts
// HighlightText가 렌더될 때마다 RegExp 생성
const safeHighlight = highlight.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
```

### 이후
```ts
// 모듈 레벨에 정의 — 앱 생애 주기 동안 한 번만 생성
const ESCAPE_REGEXP = /[.*+?^${}()|[\]\\]/g;

// HighlightText 내부
const safeHighlight = highlight.replace(ESCAPE_REGEXP, "\\$&");
```

**효과:** 고정 패턴의 RegExp는 매번 새로 컴파일하지 않고 모듈 레벨에 캐싱하면 CPU 비용이 줄어듭니다. 특히 리스트 렌더링처럼 자주 호출되는 컴포넌트에서 효과적입니다.

## 9. `server-serialization` — Footer를 RSC로 전환

**파일:** `apps/web/src/components/footer.tsx`  
**규칙:** 3.5 Minimize Serialization at RSC Boundaries (HIGH)

### 이전
```tsx
"use client";
// ... 정적 렌더링만 하는 Footer
```

### 이후
```tsx
// "use client" 제거 → React Server Component
// ... 동일한 정적 렌더링
```

**효과:** `Footer`에는 `useState`, `useEffect`, 이벤트 핸들러가 전혀 없습니다. `"use client"` 지시어를 제거하면 이 컴포넌트의 JavaScript가 클라이언트 번들에 포함되지 않아 초기 JS 페이로드가 줄어듭니다.

## 적용하지 않은 항목 (검토 후 제외)

| 규칙 | 이유 |
|---|---|
| `bundle-dynamic-imports` (framer-motion, giscus) | `optimizePackageImports`로 1차 해결. Giscus는 이미 `loading="lazy"` 설정. 추가 dynamic import는 UX 저하 가능성이 있어 보류. |
| `async-suspense-boundaries` | 현재 데이터 패칭 구조가 이미 RSC→Client Props 형태이므로 큰 환경 변경 없이 적용 어려움. 별도 리팩토링 세션 권장. |
| `server-cache-react` | Supabase 쿼리를 `React.cache()`로 감싸면 per-request dedup이 가능하나, 현재 아키텍처에서 동일 요청 내 중복 호출이 없어 우선순위 낮음. |
| `client-localstorage-schema` | localStorage 스키마 버전 관리는 현재 데이터 구조가 단순하여 도입 비용 대비 효과가 미미함. |
